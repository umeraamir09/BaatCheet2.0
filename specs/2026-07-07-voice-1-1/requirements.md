# Phase 4 — 1:1 voice (direct WebRTC): Requirements

Feature dir: `specs/2026-07-07-voice-1-1/`
Roadmap phase: **Phase 4 — 1:1 voice (direct WebRTC)** (`specs/roadmap.md`)
Mission ref: `specs/mission.md` · Stack ref: `specs/tech-stack.md`
Builds on: `specs/2026-07-06-dm-text/` (Phase 3 — the `users`/`presence`/`conversations` tables, the reactive Convex subscription + public-mutation patterns, and the 2-pane shell with the DM-selectable friends sidebar are the foundation Phase 4 hangs voice off of).

## Goal
First voice: prove direct peer-to-peer WebRTC audio between two friends, with coturn as TURN fallback. This is the roadmap's Phase 4 and the voice half of the "Talk to one friend" loop (`specs/mission.md`). A calls B from the DM thread → B sees an incoming-call toast → B accepts → two-way audio works → either side can mute/deafen/leave cleanly. The roadmap calls this "smaller surface area than group voice — good place to learn the WebRTC API," so Phase 4 uses the raw WebRTC API directly (no SFU, no wrapper library); the LiveKit SFU for group voice is Phase 6.

## In scope
- **Signaling over Convex (Decision D1):** exchange the SDP offer/answer + trickled ICE candidates via Convex docs that both sides subscribe to — reusing the Phase-2/3 reactive subscription + public-mutation pattern. No new WebSocket infra; signaling state lives in a `calls` table doc (Decision D3).
- **`calls` table in Convex (Decision D3):** a new `calls` table — `callerId`, `calleeId`, `status` (`"calling"` | `"accepted"` | `"rejected"` | `"ended"` | `"missed"`), `offerSdp`, `answerSdp` (nullable), `callerIceCandidates: string[]`, `calleeIceCandidates: string[]` (JSON-encoded `RTCIceCandidate` init dicts), `startedAt`, `connectedAt` (nullable), `endedAt` (nullable), `endReason` (nullable). Indexed by `byCallee` on `["calleeId", "startedAt"]` (incoming-call subscription) and `byCaller` on `["callerId", "startedAt"]`. Both sides subscribe to the single call doc for state transitions + ICE trickle.
- **Call lifecycle mutations (`convex/calls.ts`):** `startCall(callerId, calleeId, offerSdp)` → returns `callId`; `answerCall(callId, answerSdp)`; `rejectCall(callId)`; `endCall(callId, reason)`; `addIceCandidate(callId, side, candidate)` (appends to the caller/callee array); `markMissed(callId)` (caller-side ring timeout, ~30s). `listIncomingCalls(calleeId)` query (reactive — powers the toast) + `getCall(callId)` query (reactive — the live call-doc subscription). All public (Decision D7).
- **Raw WebRTC layer (`src/webrtc/peerConnection.ts`, Decision D4):** a thin wrapper around the browser `RTCPeerConnection` — no `simple-peer`/`peerjs` dependency. Handles `onicecandidate` → `addIceCandidate` mutation, `ontrack` → remote `<audio>` element, `createOffer`/`createAnswer`/`setLocalDescription`/`setRemoteDescription`/`addIceCandidate`, `close`. ICE servers config read from `VITE_ICE_SERVERS` (Vite-embedded JSON of `RTCIceServer[]`) — STUN + coturn TURN (Decision D2).
- **Audio capture + playback:** `getUserMedia({ audio: { echoCancellation: true, noiseSuppression: true, autoGainControl: true }, video: false })`. Remote audio played via a hidden `<audio autoplay>` element with `srcObject = remoteStream`. Mute = local audio track `enabled = false`; deafen = remote audio element muted AND local track muted (Discord semantics — you can't hear and you don't accidentally talk).
- **Call hook (`src/hooks/useCall.ts`):** owns the call state machine — `idle` → `initiating` (caller) / `ringing` (callee) → `connected` → `ended`. For the caller: `startCall` → create PC → `createOffer` → set local desc → subscribe to `getCall` → trickle ICE → wait for `answerSdp` (status → `accepted`) → set remote desc → on remote track → play. For the callee: subscribe to `listIncomingCalls` → on incoming doc → toast → accept: create PC → set remote desc (offer) → `createAnswer` → set local desc → `answerCall` → trickle ICE → on remote track → play. Leave/end → `endCall` + close PC + stop local tracks.
- **Call UI (`src/components/call/`):**
  - `IncomingCallToast.tsx` — bottom-right toast: caller avatar + "X is calling you" + Accept / Decline buttons. Rendered by `AuthenticatedLayout` when `listIncomingCalls` returns a doc.
  - `CallControls.tsx` — in-call control bar (mute, deafen, leave, live call timer, peer name/avatar). Rendered at the top of the DM thread (or as a floating overlay) while a call is active.
- **Call-initiation surface:** a call button (phone icon) in the DM thread header — clicking it starts a 1:1 voice call with the DM peer. Reuses the active DM's peer (no separate "select who to call" UI — Phase 4 calls the friend whose DM is open; the friends sidebar already lists everyone).
- **coturn provisioning (Decision D2):** a single self-hosted coturn on the Coolify VM, shared with Phase 6 (group voice + LiveKit fallback). Static long-term-mechanism credentials for v1 (Decision D5). The ICE servers JSON is provided to the frontend via `VITE_ICE_SERVERS` (env, not in the repo).

## Out of scope (deferred — explicitly NOT Phase 4)
- **Group voice / LiveKit SFU** → Phase 6 (the roadmap's "infra-heavy half"; Phase 4 is direct P2P, no SFU).
- **Video** → not in v1 (mission: pared-down; voice only for v1).
- **Screen share** → not in v1.
- **Call recording** → not in v1.
- **Call history / recent calls UI** → the `calls` docs persist (for debugging + future "recent calls") but no UI surface is built in Phase 4; a "recent calls" panel is a later polish.
- **OS notifications for incoming calls** → the Phase-4 DoD surface is the **in-app toast** (roadmap: "incoming call toast"). The tech-stack's "Tauri notification API" row is real but OS notifications are deferred to a later polish phase (the in-app toast covers the DoD; OS notifications matter when the window is unfocused, which is a background-behavior polish).
- **Time-limited / HMAC-issued coturn credentials** → deferred (Decision D5 — static credentials are the v1 stance; credential issuance via a Convex function is a later hardening pass).
- **Call quality stats / network indicators** → not in v1 (Discord-style latency bars deferred).
- **Re-ring / call-waiting** → only one active call per user at a time in v1; a second incoming call while in-call is rejected/ignored.
- **Convex auth / per-user write authorization** → deferred (Decision D7 — public call mutations are a known v1 limitation).
- **Full Discord-derived theme polish** → Phase 7 (Phase 4 call UI is functional, lightly styled — reuses the Phase-3 dark theme tokens already in place).
- **Idle CPU/RAM profiling under gaming load** → Phase 7.

## Decisions (locked for this phase)
- **D1 — Signaling channel: Convex, not a separate WebSocket.** The stack row says realtime data = Convex reactive subscriptions; WebRTC signaling (SDP offer/answer + trickled ICE) is well-served by a Convex doc both sides subscribe to. Reuses the Phase-2/3 reactive subscription + public-mutation pattern with no new infra. Cost: signaling state persists in Convex (stale call docs must be cleaned up — a client-side `endCall` + a v1-deferred cron sweep handles this; ≤10 group keeps the table tiny) and per-message latency is a bit higher than a raw WS, but call setup is not latency-critical at the signaling layer (ICE gathering dominates). A separate WS would duplicate the realtime path and add infra to operate on the Coolify VM; piggybacking on `messages` would pollute forever-retained chat history (mission v1 decision) with transient signaling noise.
- **D2 — coturn provisioned now on the Coolify VM, shared with Phase 6.** The roadmap explicitly says Phase 4 uses "coturn as TURN fallback" and the tech-stack says one self-hosted coturn is shared between 1:1 WebRTC and LiveKit fallback. Provisioning it once in Phase 4 means the DoD's "two-way audio works" passes across NAT topologies (strict/symmetric NAT where host/srflx candidates fail), and Phase 6 inherits a working TURN. Cost: a user-performed prerequisite (deploy + configure coturn on the Oracle VM, open TURN ports 3478 UDP/TCP on the firewall, issue static credentials) before the DoD can fully pass behind strict NAT. STUN-only would fail the DoD if either test client is behind strict NAT and would defer coturn to Phase 6 anyway.
- **D3 — Call state model: a dedicated `calls` table in Convex, not in-memory and not a field on `conversations`.** A `calls` doc (status + SDP offer/answer + ICE-candidate arrays + timestamps + endReason) that both sides subscribe to is the reactive source of truth — the incoming-call toast is just `listIncomingCalls(calleeId)` reacting to a new doc; state transitions (`accepted`/`rejected`/`ended`) are one mutation + one subscription update. Mirrors the Phase-3 conversation/message pattern; keeps ephemeral call state separate from forever-retained chat (mission v1). In-memory-only would require B to already be subscribed to something to get the toast and would make "missed call" invisible; piggybacking on `conversations` would couple ephemeral state to the persistent DM doc and churn a hot path. One extra table + stale-doc cleanup is the cost.
- **D4 — Raw `RTCPeerConnection`, no wrapper library.** The roadmap says Phase 4 is "a good place to learn the WebRTC API." A thin in-repo wrapper (`src/webrtc/peerConnection.ts`) over the browser `RTCPeerConnection` keeps the API explicit and adds no dependency; `simple-peer`/`peerjs` would hide exactly the surface the phase is meant to learn and would couple v1 to a third-party release cadence. The wrapper is the single place that knows about ICE servers, track enabling/disabling, and PC teardown.
- **D5 — coturn credentials: static long-term-mechanism credentials for v1.** Coturn's long-term mechanism with a shared `username` + `credential` is simple to configure and acceptable for a ≤10 trusted-friends group on a self-hosted VM. Time-limited HMAC-issued credentials (username = expiry, credential = HMAC of expiry under a shared secret) are more secure but require a backend function to mint them and add secret-handling — deferred to a later hardening pass. Known v1 limitation: anyone with the static credential can use the TURN relay; documented in `validation.md`. The credential is delivered to the client via `VITE_ICE_SERVERS` (env, not in the repo), same pattern as `VITE_DISCORD_CLIENT_ID`.
- **D6 — Incoming-call surface: in-app toast is the DoD; OS notifications deferred.** The roadmap's Phase-4 bullet says "incoming call toast" — that is the in-app toast rendered by `AuthenticatedLayout` reacting to `listIncomingCalls`. The tech-stack's "Tauri notification API" row is real but is a background-window polish (matters when the window is unfocused); it is deferred to a later phase so Phase 4 stays tight. The in-app toast fully satisfies the DoD's "B sees toast."
- **D7 — No Convex auth in v1; public call mutations, keyed by `callerId`/`calleeId`/`callId`.** Inherits Phase 2 D7 / Phase 3 D4. `startCall`, `answerCall`, `rejectCall`, `endCall`, `addIceCandidate`, `markMissed` are all public (no Convex auth middleware); the client passes its own `userId`. **Known v1 limitation:** a misbehaving client could spoof a call, hijack another user's call doc, or inject ICE candidates. Acceptable for v1 (≤10 trusted friends, Convex dev backend); hardening deferred. Flagged in `validation.md`.
- **D8 — Validation: automated gates green + manual smokes.** Lint, typecheck, `bun tauri build`, and Convex schema deploy must pass (inherited). Plus manual smokes for (i) A calls B → B sees toast → accept → two-way audio; (ii) mute/deafen round-trip; (iii) either side leaves cleanly; (iv) reject path; (v) call across NAT with coturn TURN fallback (if a test client is behind strict NAT). See `validation.md`.
- **D9 — Platform scope: Windows-only for Phase 4.** Validate on Windows (WebView2) only. No macOS `NSMicrophoneUsageDescription`, no Linux WebKitGTK permission handling. Cross-platform mic hardening defers. WebView2 mic access is governed by Windows OS-level mic privacy; verify on first smoke.
- **D10 — ICE env shape: single `VITE_ICE_SERVERS`, remove the separate vars.** Remove `VITE_TURN_URL`/`VITE_TURN_USERNAME`/`VITE_TURN_PASSWORD`/`VITE_STUN_URL` from `.env.example`; add one `VITE_ICE_SERVERS=` line with the JSON example. Code reads one var; falls back to Google STUN if unset.
- **D11 — Busy + offline callee handling: disable call button if peer offline; auto-reject + hide toast if I'm in-call.** Call button gates on `peer.online === true`. Incoming call while I'm in-call: hide toast AND auto-call `rejectCall` so the caller gets a fast busy signal (not a 30s `markMissed` wait).
- **D12 — Call UI placement: floating overlay owned by `AuthenticatedLayout`.** `CallControls` + `<audio>` render in `AuthenticatedLayout` as a floating bar over the main pane, independent of the active DM. User can browse DMs mid-call. Toast also lives here. `DMThread` gets a call button + a `startCallWithPeer` callback prop.

## Context

The hard constraint ("idle at low CPU/RAM while a game is running, voice must not stutter") shapes the shell but is *measured* in Phase 7, not Phase 4; Phase 4 only needs to *work*. ≤10-person trusted-friends group — informs D5 (static TURN creds) and D7 (public mutations).
- `specs/tech-stack.md` — Voice — 1:1 calls row: direct WebRTC P2P, no server relay for 2 people. STUN/TURN row: single self-hosted coturn shared between 1:1 WebRTC and LiveKit fallback (D2). Native notifications row exists but Phase 4 wires only the in-app toast (D6). Deployment target row: Oracle VM behind Coolify — coturn deploys there (D2, user-performed).
- `specs/roadmap.md` Phase 3 — established `conversations` + `messages` + `typing`, the reactive subscription + public-mutation pattern, the 2-pane shell with the DM-selectable friends sidebar, and the `getMyUser`-derived `users._id` the call hook uses as `callerId`/`calleeId`. Phase 4 adds `calls` and hangs a call button off the DM thread header.
- `specs/roadmap.md` Phase 4 DoD: *A calls B → B sees toast → accept → two-way audio works → either side can mute/deafen/leave cleanly.*
- `convex/schema.ts` (current) — declares `users`, `presence`, `conversations`, `messages`, `typing`, and the `counter` remnant. Phase 4 adds `calls`; existing tables and `counter` are untouched.
- `src/components/AuthenticatedLayout.tsx` (current) — Phase-3 2-pane layout, owns `usePresence`, the active-DM state, and the Tauri `onCloseRequested` teardown. Phase 4 extends it to subscribe to `listIncomingCalls`, render the toast, render the in-call control bar, and tear down any active call on logout / window close.
- `src/components/DMThread.tsx` + `src/hooks/useDMThread.ts` (current) — the DM thread the call button lives in; the call hook reuses the active DM's peer profile for the call UI.
- `src/auth.ts` + `src/hooks/useAuth.ts` (current) — the `user.id` (Discord user id) and the Phase-2 `getMyUser`-derived `users._id` are the call identities.

## User-performed prerequisites (not agent-executable)
- **Phase 3 must be landed** (present on the working branch) so `users`/`presence`/`conversations`/`messages`, the reactive subscription pattern, the `getMyUser` query, the 2-pane shell, and the DM-selectable sidebar are available. Phase 4 branches from the Phase-3 tip.
- **Deploy + configure coturn on the Oracle VM behind Coolify (Decision D2):** install coturn, enable the long-term mechanism, set a static `username` + `credential` (Decision D5), open firewall ports 3478 UDP/TCP (and 5349 TCP for TLS if used), and confirm the TURN server is reachable from the public internet. The agent cannot deploy infra on the VM. Provide the resulting ICE servers JSON to the frontend via `VITE_ICE_SERVERS` in `.env.local` (e.g. `VITE_ICE_SERVERS='[{"urls":"stun:stun.l.google.com:19302"},{"urls":"turn:<vm-host>:3478","username":"baatcheet","credential":"<secret>"}]'`).
- **The Convex dev backend (`bunx convex dev`) must be running** (or re-run after schema changes) so `_generated/` is regenerated for the new `calls` table. The agent runs this; if the Convex deployment is unreachable, the user must confirm the dev backend / credentials.
- **Two Discord test accounts on different networks (for the cross-NAT smoke)** — the user must have (or create) a second Discord account to validate the live call DoD. To exercise the coturn TURN fallback specifically (smoke 5), the two accounts should be on networks where direct UDP fails (e.g. one on a strict-NAT mobile hotspot, one on home wifi). The agent cannot create Discord accounts or control network topologies.
