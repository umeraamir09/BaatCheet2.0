## 2. Convex token-mint action (`convex/livekit.ts`) (Decisions D3, D11, D13, D16)
- 2.1 Create `convex/livekit.ts` with a `mintToken(userId)` **internal action** (not a mutation/query — `livekit-server-sdk` uses Node APIs that Convex actions allow):
  - Load the `users` doc for `args.userId` (`ctx.db.get(args.userId)`); throw `ConvexError("User not found")` if missing.
  - Build the identity/name/metadata per Decision D11: `identity = args.userId` (the `users._id` string), `name = user.displayName ?? user.username`, `metadata = JSON.stringify({ avatarUrl: user.avatarUrl, displayName: user.displayName, username: user.username })`.
  - Use `livekit-server-sdk`'s `AccessToken` with `process.env.LIVEKIT_API_KEY` + `process.env.LIVEKIT_API_SECRET`; `token.addGrant({ room: "lobby", roomJoin: true, canPublish: true, canSubscribe: true, canPublishData: false })`; `token.identity = identity`; `token.name = name`; `token.metadata = metadata`; `const jwt = await token.toJwt()`.
  - Return `{ token: jwt, url: process.env.VITE_LIVEKIT_URL ?? "" }` — OR return only the token and let the frontend read `VITE_LIVEKIT_URL` from `import.meta.env`. **Decision: return only `{ token }`**; the frontend reads the URL from Vite env (keeps the action server-pure; the URL is a public frontend value, not a secret). Document this in a comment.
  - Public (Decision D13 — no Convex auth middleware). Document the v1 spoofing limitation in the file header comment matching `convex/calls.ts`/`convex/lobby.ts` posture: "a misbehaving client could pass another user's `userId` and join as them; acceptable for v1 (≤10 trusted friends); hardening deferred."
- 2.2 Run `bunx convex dev` (or `bunx convex codegen`) to register the new `livekit.ts` action in `_generated/`; confirm `api.livekit.mintToken` resolves in the frontend. If `livekit-server-sdk` fails to load in the Convex action runtime, document the error and fall back to hand-rolling the LiveKit JWT (HS256, `iss` = API key, `sub` = identity, `video: { room: "lobby", roomJoin: true, canPublish: true, canSubscribe: true }` grant) using Convex's available crypto — but prefer the official SDK first.
- 2.3 Confirm the user has set `LIVEKIT_API_KEY` + `LIVEKIT_API_SECRET` in the Convex deployment env (`bunx convex env add`). If the action throws "API key/secret not set," that's a user-performed-prereq failure (the agent surfaces it, doesn't fix it).
- **Done:** `convex/livekit.ts` exposes the `mintToken` action; the v1 spoofing limitation is documented; `_generated/` regenerated; lint/typecheck clean; no schema change (D16).

## 3. Group-voice hook (`src/hooks/useGroupVoice.ts`) (Decisions D2, D4, D5, D7, D9, D11, D12)
- 3.1 Create `src/hooks/useGroupVoice.ts`. State machine: `disconnected` → `connecting` (mint token + `room.connect`) → `connected` → `disconnected`. Expose:
  ```
  { connected: boolean; connecting: boolean; error: string | null;
    muted: boolean; deafened: boolean;
    join: () => Promise<void>; leave: () => Promise<void>;
    setMuted: (m: boolean) => void; setDeafened: (d: boolean) => void;
    roomRef: React.MutableRefObject<Room | null>; // for the VoiceStage to attach hooks
    token: string | null; } // the VoiceStage's <LiveKitRoom> needs the token
  ```
  The hook does NOT render `<LiveKitRoom>` — it owns the `Room` lifecycle + mute/deafen; the `VoiceStage` component (task 4) wraps the roster in `<LiveKitRoom>` using the token + URL from the hook (Decision D2 — the React hooks need a `<LiveKitRoom>` context provider). **Reconcile in 3.2:** either the hook owns the `<LiveKitRoom>` provider OR the `VoiceStage` does — pick the one that keeps the roster hooks (`useParticipants`) inside the provider. Plan fix: the `VoiceStage` renders `<LiveKitRoom token={token} serverUrl={url} connect={true}>` and the roster inside it; the hook owns `join`/`leave`/mute/deafen + the `Room` instance via `livekit-client`'s `room` object passed through. (Use `@livekit/components-react`'s `useRoom` inside the provider for the room object, OR the hook creates the `Room` directly via `new Room(...)` + `room.connect()` and passes it to `<LiveKitRoom room={room}>`. Prefer the latter — the hook owns the lifecycle, the component owns the rendering.)
- 3.2 `join()` (Decision D5 — single-click):
  - If `connected` or `connecting`, return (idempotent).
  - **Mutual exclusivity (Decision D9):** accept an optional `onNeedLeave1to1: () => Promise<void>` callback (wired from `AuthenticatedLayout` to `call.leave()`); call + `await` it BEFORE proceeding if a 1:1 call is active. (The hook doesn't import `useCall` — the layout injects the leave callback to avoid a circular hook dependency.)
  - `setConnecting(true)` → `const { token } = await mintTokenMutation({ userId: myUserId })` → create `const room = new Room({ adaptiveStream: true, dynacast: true, audioCaptureDefaults: { echoCancellation: true, noiseSuppression: true, autoGainControl: true } })` → `await room.connect(VITE_LIVEKIT_URL, token)` → `await room.localParticipant.setMicrophoneEnabled(true)` (enable mic on join) → `setConnected(true)`, store `room` in `roomRef`.
  - On `RoomEvent.ParticipantConnected`/`ParticipantDisconnected`/`Disconnected`/`Reconnecting` → update state (the roster reads these via the React hooks, so explicit state may be minimal — the hooks re-render on room events).
  - On `RoomEvent.Disconnected` (unexpected) → `setConnected(false)`, cleanup.
- 3.3 `leave()` (Decision D5 — one-click): if `roomRef.current`, `await room.disconnect()` → stop local tracks (`room.localParticipant.audioTrackPermissions` / iterate `localParticipant.trackPublications` and `track.stop()`) → `roomRef.current = null` → `setConnected(false)` + reset mute/deafen. Idempotent + race-timeout-guarded (2s) like `useCall.leave`.
- 3.4 `setMuted(bool)` (Decision D7): `roomRef.current?.localParticipant.setMicrophoneEnabled(!muted)`. `setDeafened(bool)`: mute local mic (same as mute) + mute all remote audio — iterate `room.remoteParticipants`, for each audio track `track.setEnabled(!deafened)` (or set the master `<audio>` output element muted — the `VoiceStage` hosts the audio elements via `@livekit/components-react`'s `AudioRenderer`/`useAudioRenderer`; deafen toggles their `muted`). Discord semantics: deafen = can't hear AND can't accidentally talk (inherited from Phase 4 D3).
- 3.5 Teardown on unmount: a `useEffect` with `[]` deps that calls `leave()` on unmount if `connected` (the same pattern as `useCall`'s unmount effect — guarded by a status ref so it only fires on real unmount).
- 3.6 Read `VITE_LIVEKIT_URL` from `import.meta.env.VITE_LIVEKIT_URL`; if unset, `join()` throws a clear error ("VITE_LIVEKIT_URL not set — LiveKit server not configured"). Fall back is NOT an option for group voice (unlike Phase 4's public-STUN fallback — there's no "public LiveKit").
- 3.7 Mute/deafen state lifted to the hook so the `VoiceStage` UI reflects it.
- **Done:** the hook owns the full group-voice lifecycle; `join` mints + connects; `leave` disconnects + stops tracks; mute/deafen round-trip; teardown on unmount; mutual-exclusivity callback wired; lint/typecheck clean.

## 6. AuthenticatedLayout integration + mutual exclusivity + teardown (Decisions D9, D12)
- 6.1 Instantiate `const groupVoice = useGroupVoice(presence.userId)` in `AuthenticatedLayout` (alongside `useCall`).
- 6.2 **Mutual exclusivity wiring (Decision D9):**
  - `useGroupVoice.join` needs to leave a 1:1 call first: pass `onNeedLeave1to1` to `join` (or wire via the layout) — when `call.status !== "idle" && call.status !== "ended"`, `await call.leave("left")` before `groupVoice.join()` proceeds. Implement as: the `joinVoice` callback the layout passes to `LobbyThread`/`VoiceStage` wraps `groupVoice.join` with the 1:1-leave-first check.
  - `useCall.startCall` / `accept` need to leave group voice first: wrap the layout's `startCallWithPeer` (passed to `DMThread`) — if `groupVoice.connected`, `await groupVoice.leave()` before `call.startCall(...)`. Same for the accept path — the layout's `call.accept` wrapper checks `groupVoice.connected` first. (The layout owns the composition, so the wrappers live there — the hooks stay decoupled per 3.2.)
- 6.3 Render the side-by-side lobby (task 5.2) when in lobby view + group voice active. The floating 1:1 `CallControls` + `IncomingCallToast` persist across view modes (Phase 5 D8) AND coexist with the `VoiceStage` — but per D9 they're never both active (mutual exclusivity), so at most one of {1:1 call overlay, group voice stage} is visible at a time. Confirm the floating overlay still renders over the lobby when a 1:1 call is active (unchanged from Phase 5).
- 6.4 **Teardown (Decision D12):** extend `handleLogout` + the Tauri `onCloseRequested` teardown:
  - `handleLogout`: insert `if (groupVoice.connected) { await groupVoice.leave(); }` BEFORE the existing `if (call.status !== idle) { await call.leave("left"); }` + `await presence.goOffline()` sequence. Ordering: group-voice leave → 1:1 call leave → goOffline (only one of the two voice paths is ever active per D9, but the ordering is safe either way).
  - `onCloseRequested`: add the group-voice disconnect to the existing race-timeout teardown — `if (groupVoice.connected) { await Promise.race([groupVoice.leave(), new Promise(r => setTimeout(r, 3000))]); }` before the 1:1 call leave + `goOffline` steps.
- 6.5 No view-mode change on join/leave — group voice is independent of the lobby/DMs view mode (you can be in voice while viewing a DM; the `VoiceStage` is lobby-view-only, but the audio persists across view switches — the `Room` lives in the hook, not the component). Confirm: switching to "DMs" while in group voice keeps the audio connected (the `VoiceStage` unmounts but `groupVoice.connected` stays true + audio keeps playing); switching back to "Lobby" remounts the `VoiceStage` and re-renders the roster. (This is the LiveKit equivalent of Phase 5 D8's "floating call overlay persists across view modes.")
- 6.6 Logout clears `VIEW_MODE_KEY` + `ACTIVE_DM_KEY` (unchanged from Phase 5). Group voice has no localStorage pref (no "last voice state" to persist — joining is an explicit action, not a restored state).
- **Done:** the layout composes `useGroupVoice` + the side-by-side lobby; mutual exclusivity is enforced (join voice leaves 1:1, start 1:1 leaves voice); teardown disconnects group voice on logout + window close before `goOffline`; the 1:1 overlay still works; lint/typecheck clean.
