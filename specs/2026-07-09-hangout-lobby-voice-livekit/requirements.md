# Phase 6 ‚Äî Hangout lobby (group voice via LiveKit): Requirements

Feature dir: `specs/2026-07-09-hangout-lobby-voice-livekit/`
Roadmap phase: **Phase 6 ‚Äî Hangout lobby (group voice via LiveKit)** (`specs/roadmap.md`)
Mission ref: `specs/mission.md` ¬∑ Stack ref: `specs/tech-stack.md`
Builds on: `specs/2026-07-07-voice-1-1/` (Phase 4 ‚Äî the raw-WebRTC `PeerCall` + `useCall` state machine + `CallControls`/`IncomingCallToast` + the floating-overlay + teardown-ordering patterns + the coturn-on-Coolify prerequisite), `specs/2026-07-08-hangout-lobby-text/` (Phase 5 ‚Äî the single shared `conversations` lobby doc with `key: "group:lobby"`, the icon rail + view-mode state, the `LobbyThread` that Decision D8 left call-button-less, and the deferred side-by-side voice/text layout), and Rich messaging (the shared `MessageBubble`/`Composer` the lobby text half uses).

## Goal
The voice half of the centerpiece: one always-open group voice room for the whole friend group, backed by a self-hosted LiveKit SFU. Any member clicks "Join Voice" in the lobby ‚Üí dropped straight into the in-progress call (no invite/accept dance); mute/deafen/leave reuse the Phase-4 semantics; leaving and rejoining is one click with no call-teardown dance. This is the roadmap's Phase 6 and the voice half of the "Talk to everyone" loop (`specs/mission.md`). The roadmap calls this "the infra-heavy half; saved for when the rest already works," and the tech-stack row says it explicitly: mesh WebRTC "breaks past a handful of peers," so group voice uses the LiveKit SFU (self-hosted on the Coolify VM alongside Convex + coturn) rather than the Phase-4 direct-P2P approach. Phase 6 also lands the side-by-side lobby layout Phase 5 D8 deferred ("Phase 6 lands when group voice lands") and the "Join Voice" button Phase 5 D8 left out of the `LobbyThread` header.

## In scope
- **LiveKit SFU, not mesh WebRTC (Decision D1):** a self-hosted LiveKit server on the Coolify VM (same VM as Convex + coturn) is the group voice backend. The single shared voice room is named `"lobby"` (mirrors the text lobby's `key: "group:lobby"`). LiveKit auto-starts the room on the first participant connect and auto-empties it on the last disconnect ‚Äî the room is conceptually "always open" (anyone can join anytime), no create/destroy lifecycle to manage. This is the infra-heavy piece the roadmap saved for last; the agent writes the client + token-mint, the user deploys the LiveKit server (user-performed prerequisite, like Phase 4's coturn).
- **Official LiveKit SDK, not a raw reimplementation (Decision D2):** `livekit-client` (core JS SDK) + `@livekit/components-react` (React hooks: `useRoom`, `useLocalParticipant`, `useParticipants`) for the frontend, and `livekit-server-sdk` (token minting) in the Convex action. Unlike Phase 4's "learn the raw WebRTC API" stance (D4), LiveKit's value IS the SDK ‚Äî re-implementing the SFU signaling protocol would be foolish and would couple v1 to a third-party release cadence in the worst way. The tech-stack explicitly says LiveKit "ships official React client SDKs." We use the hooks to build our own pared-down `VoiceStage`; the prebuilt video-conference components are not used (video is out of scope for v1, and the prebuilts are video-focused/opinionated).
- **Token minting via a Convex action; API secret stays server-side (Decision D3):** a Convex action `convex/livekit.ts` ‚Üí `mintToken(userId)` reads `LIVEKIT_API_KEY` + `LIVEKIT_API_SECRET` from the Convex deployment environment (set via `bunx convex env add`, NOT Vite env) and mints a LiveKit JWT access token for room `"lobby"` with `identity = users._id`, `name = displayName ?? username`, and `metadata` = JSON `{ avatarUrl, displayName, username }` (Decision D11). The frontend receives only the token string + reads `VITE_LIVEKIT_URL` (Vite env). The API secret NEVER reaches the client. This corrects the `.env.example` flat-listing of `LIVEKIT_API_KEY`/`LIVEKIT_API_SECRET` ‚Äî Phase 6 clarifies they are Convex-server env, not frontend env.
- **Single fixed room `"lobby"`; LiveKit is the source of truth for voice state (Decision D4):** no Convex doc tracks the voice room ‚Äî LiveKit's room state IS the source of truth (participant list, mute states, speaking indicators). This contrasts with Phase 4's `calls` table (where Convex was the signaling channel); for group voice, LiveKit owns the realtime voice state and the client subscribes via the SDK's event-driven hooks. No new Convex table for voice room state (no schema change ‚Äî D16).
- **Single-click join; no invite/accept (Decision D5):** a "Join Voice" button in the `LobbyThread` header (the surface Phase 5 D8 explicitly deferred). Click ‚Üí `mintToken` action ‚Üí `room.connect(VITE_LIVEKIT_URL, token)` ‚Üí you're in. Leaving is one click ("Leave Voice" ‚Üí `room.disconnect()` + stop local tracks). There is NO incoming-call toast / invite / accept flow for group voice ‚Äî the room is always open; you join, you don't get "called" (contrast with 1:1 Phase 4's toast). Other members see the new participant appear in the roster live via the SDK.
- **Side-by-side lobby layout: voice left, text right (Decision D6 ‚Äî the deferred Phase-5 D8 layout):** when connected to group voice, the lobby view splits inside `<main>`: a `VoiceStage` (left, fixed ~300px) showing the participant roster + mute/deafen/leave controls, and the `LobbyThread` (right, `flex-1`) for text. When NOT in voice, the lobby is full-width text (current Phase-5 behavior, unchanged). The split is owned by a new `LobbyView` component (or by `AuthenticatedLayout`'s main-pane conditional in lobby mode ‚Äî the plan fixes the ownership).
- **Reuse mute/deafen/leave semantics, separate component (Decision D7):** the Phase-4 `CallControls` is 1:1-specific (single peer name/avatar, call timer, "Connected to X"). Group voice has a roster, not a single peer. A separate `VoiceStage` component reuses the mute/deafen/leave button iconography + semantics (mute = local mic track disabled; deafen = remote audio output muted AND local mic muted ‚Äî Discord semantics, inherited from Phase 4) but renders the LiveKit participant roster + speaking indicators instead of a single peer. `CallControls` stays unchanged for 1:1.
- **Participant roster from LiveKit room state (Decision D8):** the `VoiceStage` renders the live participant list (local + remote) via `@livekit/components-react`'s `useParticipants`. Each entry: avatar (from the token metadata ‚Äî D11), name (`displayName ?? username`), a speaking indicator (green ring when `isSpeaking`), and a mute icon when the participant's mic is disabled. This is the "see who's around" in the voice context ‚Äî the Phase-5-deferred "lobby member roster panel" is satisfied by the voice roster (a roster is only meaningful when voice is active; a separate always-on member list is Discord chrome the pared-down mission doesn't need).
- **Speaking indicator (Decision D8):** LiveKit exposes `isSpeaking` on each participant (VAD-driven). The `VoiceStage` renders a green ring/glow around the avatar of the active speaker. No VAD-threshold tuning UI for v1 (LiveKit defaults).
- **Mutual exclusivity: one active voice connection per user (Decision D9):** joining group voice leaves any active 1:1 call; starting a 1:1 call (Phase 4) leaves the group voice. Rationale: single mic resource, simplicity, Discord-parity behavior (joining a voice channel drops you out of any other voice connection). Enforced in the hooks: `useGroupVoice.join()` calls `call.leave()` first if `call.status` is active; `useCall.startCall()`/`accept()` calls `groupVoice.leave()` first if `groupVoice.connected` is true. One mic, one voice connection.
- **coturn as LiveKit TURN fallback (Decision D10 ‚Äî server config, not client code):** LiveKit's server config points at the existing coturn (the Phase-4 D2 shared coturn on the Coolify VM) as its TURN server. Participants behind strict NAT relay through coturn. This is a LiveKit server config concern (user-performed prerequisite); the client just connects to the LiveKit URL ‚Äî no `VITE_ICE_SERVERS` involvement for group voice (LiveKit manages its own ICE). The Phase-4 coturn is now shared across 1:1 WebRTC, LiveKit fallback, and LiveKit's own TURN ‚Äî one coturn, three uses (tech-stack: "single self-hosted coturn, shared").
- **Identity + metadata in the LiveKit token (Decision D11):** `identity = users._id` (Convex id ‚Äî consistent with the rest of the app's identity model: presence, messages, calls all key on `users._id`); `name = displayName ?? username`; `metadata` = JSON string `{ avatarUrl, displayName, username }` so the roster renders without a separate Convex lookup (the `VoiceStage` parses `participant.metadata`). The token is scoped to room `"lobby"` only (no cross-room access).
- **Teardown: disconnect on logout + window close (Decision D12):** reuse the Phase-4 `onCloseRequested` + `handleLogout` teardown ordering ‚Äî insert `groupVoice.leave()` BEFORE `goOffline()` (alongside the existing 1:1 `call.leave()`). `room.disconnect()` + stop local tracks + release `getUserMedia`. The Tauri close teardown gains a group-voice disconnect step (with the same 3s race-timeout backstop the 1:1 call uses).
- **Group-voice hook (`src/hooks/useGroupVoice.ts`):** owns the group voice state machine ‚Äî `disconnected` ‚Üí `connecting` (mint token + connect) ‚Üí `connected` ‚Üí `disconnected`. Exposes `{ connected, connecting, participants, localParticipant, join, leave, setMuted, setDeafened, muted, deafened, room }`. The hook wraps a `livekit-client` `Room` instance (created on `join`, disconnected on `leave`/unmount/logout). Mute via `localParticipant.setMicrophoneEnabled(bool)`; deafen via muting the room's remote audio output + `setMicrophoneEnabled(false)`. The hook does NOT subscribe to a Convex doc for voice state ‚Äî LiveKit's event-driven room state is the subscription (D4).
- **`convex/livekit.ts` ‚Äî the token-mint action (Decision D3, D13):** `mintToken(userId)` internal action: load the `users` doc (for `displayName`/`username`/`avatarUrl`), use `livekit-server-sdk`'s `AccessToken` to mint a token for room `"lobby"` with the identity/name/metadata per D11, return the token string. Public (Decision D13 ‚Äî no Convex auth; v1 spoofing limitation accepted). The action reads `LIVEKIT_API_KEY`/`LIVEKIT_API_SECRET` from `process.env` (Convex deployment env).
- **Env reconciliation (Decision D3):** `.env.example` keeps `VITE_LIVEKIT_URL` (frontend) but re-comment `LIVEKIT_API_KEY`/`LIVEKIT_API_SECRET` as "Convex server env ‚Äî set via `bunx convex env add`, NOT `.env.local`." The user sets the two secrets in the Convex deployment environment. `src/vite-env.d.ts` gains `readonly VITE_LIVEKIT_URL?: string;`.

## Out of scope (deferred ‚Äî explicitly NOT Phase 6)
- **Video** ‚Üí not in v1 (mission: pared-down; voice only for v1). LiveKit handles video, but Phase 6 publishes audio tracks only.
- **Screen share** ‚Üí not in v1.
- **Call / voice recording** ‚Üí not in v1.
- **Push-to-talk** ‚Üí not in v1 (mission: pared-down; open-mic + LiveKit default VAD).
- **Voice activity detection threshold tuning UI** ‚Üí LiveKit default VAD; no per-user threshold slider for v1.
- **Multiple voice rooms / channels** ‚Üí not in v1 (mission: "one single shared space ‚Äî no channel switching"). Single `"lobby"` room.
- **Voice permissions / server-mute-others / kick** ‚Üí not in v1 (‚â§10 trusted friends; no moderation tooling ‚Äî mission non-goals: "roles, permissions, moderation tooling").
- **1:1 voice via LiveKit** ‚Üí Phase 4's direct WebRTC stays for 1:1 (no reason to route 2 people through an SFU ‚Äî tech-stack: "no server relay needed for 2 people"). LiveKit is group-only.
- **Incoming-call toast / invite flow for group voice** ‚Üí explicitly NOT Phase 6 (Decision D5 ‚Äî the room is always open; you join, you don't get "called"). The Phase-4 toast is 1:1-only.
- **OS notifications for voice events (who joined/left)** ‚Üí out of scope for v1 (mission: offline push deferred; in-app roster is the surface).
- **Convex auth / per-user write authorization for token minting** ‚Üí deferred (Decision D13 ‚Äî public token-mint action is a known v1 limitation).
- **Full Discord-derived theme polish** ‚Üí Phase 7 (Phase 6 voice UI is functional, lightly styled ‚Äî reuses the Phase-3/4 dark theme tokens already in place).
- **Idle CPU/RAM profiling under gaming load + formal audio-stutter-under-load measurement** ‚Üí Phase 7. Phase 6 includes only a light "audio stable with 3+ participants" smoke (the DoD); the formal under-load profiling is Phase 7 (the hard constraint's measurement phase).
- **LiveKit server-side recording / egress / insights** ‚Üí not in v1.

## Decisions (locked for this phase)
- **D1 ‚Äî LiveKit SFU, not mesh WebRTC. Single room `"lobby"`, always-open.** Mesh WebRTC breaks past a handful of peers (tech-stack); LiveKit is the self-hosted SFU on the Coolify VM (same VM as Convex + coturn). The single room `"lobby"` mirrors the text lobby's `key: "group:lobby"` ‚Äî one shared space, no channel fragmentation (mission). LiveKit auto-starts the room on first connect and auto-empties on last disconnect; there is no create/destroy lifecycle to manage and no Convex doc tracking the room (D4). Cost: a user-performed prerequisite (deploy + configure LiveKit on the VM, open ports, set API key/secret in Convex env) before the DoD can pass ‚Äî the same posture as Phase 4's coturn. The alternative ‚Äî mesh WebRTC scaling to ‚â§10 ‚Äî is exactly what the tech-stack rejects ("mesh breaks past a handful of peers"), and reusing Phase 4's `PeerCall` for an N-way mesh would duplicate the WebRTC surface for no benefit.
- **D2 ‚Äî Official LiveKit SDK (`livekit-client` + `@livekit/components-react` + `livekit-server-sdk`), not a raw reimplementation.** Phase 4's "learn the raw WebRTC API" stance (D4) does NOT carry to group voice ‚Äî LiveKit's value IS the SDK (signaling, reconnection, simulcast, participant management, VAD). Re-implementing the SFU client protocol would be a large, fragile surface coupled to LiveKit's release cadence. The tech-stack explicitly says LiveKit "ships official React client SDKs." We use the React hooks (`useRoom`, `useLocalParticipant`, `useParticipants`) to build our own pared-down `VoiceStage`; the prebuilt video-conference components are not used (video is out of scope + the prebuilts are video-focused/opinionated). This adds 3 new runtime dependencies ‚Äî the first since Phase 4 (which added none). Justified by the tech-stack row; recorded in `validation.md` (the Phase-5 "no new dependencies" gate does NOT apply to Phase 6).
- **D3 ‚Äî Token minting via a Convex action; `LIVEKIT_API_SECRET` stays server-side.** A Convex action (`convex/livekit.ts` ‚Üí `mintToken`) uses `livekit-server-sdk`'s `AccessToken` with `LIVEKIT_API_KEY` + `LIVEKIT_API_SECRET` read from the Convex deployment environment (`process.env`, set via `bunx convex env add`) to mint a JWT scoped to room `"lobby"`. The frontend receives only the token string + reads `VITE_LIVEKIT_URL` (Vite env). The API secret NEVER reaches the client (unlike `VITE_ICE_SERVERS` which carries the coturn credential ‚Äî that's a TURN relay credential, lower-stakes; the LiveKit API secret grants token-minting authority and must stay server-side). This corrects the `.env.example` flat-listing of `LIVEKIT_API_KEY`/`LIVEKIT_API_SECRET` ‚Äî Phase 6 clarifies they are Convex-server env. Cost: the user must set two secrets in the Convex deployment (not `.env.local`); a misconfigured deployment is a user-performed-prereq failure, not a code bug. The alternative ‚Äî minting tokens client-side with the secret in Vite env ‚Äî would expose the API secret to every user and let anyone mint arbitrary tokens; rejected.
- **D4 ‚Äî LiveKit is the source of truth for voice room state; no Convex table for voice state.** The participant list, mute states, speaking indicators, and connection state all live in LiveKit's room state, which the client subscribes to via the SDK's event-driven hooks. No `voiceRooms`/`voiceParticipants` Convex table, no schema change (D16). This contrasts with Phase 4's `calls` table (where Convex was the signaling channel for 1:1) ‚Äî group voice's realtime state is high-frequency (speaking/mute toggles) and is exactly what the SFU's signaling already carries; mirroring it into Convex would double the realtime path and thrash a hot table. Cost: voice room state is not queryable from Convex (no "who's in voice right now" Convex query) ‚Äî acceptable for v1 (the roster is a live UI surface, not a persisted record). A future "who was in voice" history feature would need a separate persisted table; deferred.
- **D5 ‚Äî Single-click join; no invite/accept. The room is always open.** A "Join Voice" button in the `LobbyThread` header (the surface Phase 5 D8 explicitly deferred: "group voice is Phase 6"). Click ‚Üí `mintToken` ‚Üí `room.connect()` ‚Üí you're in. No incoming-call toast, no invite, no accept ‚Äî the room is always open (mission: "group voice room always 'open'", roadmap: "single-click join drops you into the in-progress call"). Leaving is one click ("Leave Voice" ‚Üí `room.disconnect()`). Other members see the new participant appear in their roster live via the SDK. This contrasts with Phase 4's 1:1 toast/accept flow ‚Äî group voice is join-oriented, not call-oriented. The alternative ‚Äî an invite/accept flow for group voice ‚Äî would add a `calls`-table-style signaling layer for no benefit (the room is open; just join).
- **D6 ‚Äî Side-by-side lobby layout: voice left, text right (the deferred Phase-5 D8 layout).** When connected to group voice, the lobby view splits inside `<main>`: `VoiceStage` (left, fixed ~300px) + `LobbyThread` (right, `flex-1`). When NOT in voice, the lobby is full-width text (current Phase-5 behavior, unchanged). The split is owned by `AuthenticatedLayout`'s main-pane conditional in lobby mode (it already owns the `viewMode` + `lobbyDoc` + `presence.userId` wiring ‚Äî D6 ownership is fixed in the plan). The alternative ‚Äî `LobbyThread` rendering the voice stage internally ‚Äî would couple the text thread to voice and make the no-voice path conditional; keeping them as siblings composed by `AuthenticatedLayout` is cleaner and matches the existing icon-rail/sidebar/main composition.
- **D7 ‚Äî Reuse mute/deafen/leave semantics; separate `VoiceStage` component.** The Phase-4 `CallControls` is 1:1-specific (single peer name/avatar, call timer, "Connected to X"). Group voice has a roster, not a single peer. A separate `VoiceStage` reuses the mute/deafen/leave button iconography + Discord semantics (mute = local mic disabled; deafen = remote audio output muted AND local mic muted) but renders the roster + speaking indicators. `CallControls` stays unchanged for 1:1. The alternative ‚Äî generalizing `CallControls` with peer-vs-group branches ‚Äî would couple two distinct UIs (1:1 call bar vs. group voice stage) and make both harder to read, the same reasoning as Phase 5 D5 (separate `LobbyThread` vs. generalized `DMThread`).
- **D8 ‚Äî Participant roster from LiveKit room state; speaking indicator via `isSpeaking`.** The `VoiceStage` renders local + remote participants via `@livekit/components-react`'s `useParticipants`. Each entry: avatar (from token metadata ‚Äî D11), name (`displayName ?? username`), green ring/glow when `isSpeaking` (LiveKit VAD), mute icon when mic disabled. This is the voice-context "see who's around" ‚Äî the Phase-5-deferred "lobby member roster panel" is satisfied by the voice roster (a roster is meaningful when voice is active; a separate always-on member list is Discord chrome the pared-down mission rejects). No VAD-threshold UI for v1 (LiveKit defaults).
- **D9 ‚Äî Mutual exclusivity: one active voice connection per user.** Joining group voice leaves any active 1:1 call; starting/accepting a 1:1 call leaves the group voice. Rationale: single mic resource (one `getUserMedia` handle), simplicity, Discord-parity (joining a voice channel drops you out of any other voice connection). Enforced in the hooks: `useGroupVoice.join()` calls `call.leave()` first if `call.status` is active; `useCall.startCall()`/`accept()` calls `groupVoice.leave()` first if `groupVoice.connected`. One mic, one voice connection. The alternative ‚Äî allowing simultaneous 1:1 + group voice ‚Äî would require audio mixing/selection UI and two `getUserMedia` streams (or a shared stream with complex routing), well beyond v1's pared-down scope.
- **D10 ‚Äî coturn as LiveKit TURN fallback (server config, not client code).** LiveKit's server config points at the existing Phase-4 coturn on the Coolify VM as its TURN server. Participants behind strict NAT relay through coturn. This is a LiveKit server config concern (user-performed prerequisite); the client just connects to `VITE_LIVEKIT_URL` ‚Äî no `VITE_ICE_SERVERS` involvement for group voice (LiveKit manages its own ICE). The Phase-4 coturn is now shared across 1:1 WebRTC, LiveKit fallback, and LiveKit's own TURN ‚Äî one coturn, three uses (tech-stack: "single self-hosted coturn, shared"). Cost: the user configures LiveKit's TURN block to point at coturn (a server-config line, not client code); if misconfigured, strict-NAT participants fail to connect (the conditional smoke 6 catches this).
- **D11 ‚Äî Identity + metadata in the LiveKit token.** `identity = users._id` (Convex id ‚Äî consistent with presence/messages/calls all keying on `users._id`); `name = displayName ?? username`; `metadata` = JSON string `{ avatarUrl, displayName, username }` so the roster renders without a separate Convex lookup (the `VoiceStage` parses `participant.metadata`). The token is scoped to room `"lobby"` only. The alternative ‚Äî identity = `discordId` ‚Äî would diverge from the app's `users._id` identity model and require a lookup layer; `users._id` is already what the frontend has in hand from `usePresence`.
- **D12 ‚Äî Teardown: disconnect on logout + window close, before `goOffline`.** Reuse the Phase-4 `onCloseRequested` + `handleLogout` teardown ordering ‚Äî insert `groupVoice.leave()` BEFORE `goOffline()` (alongside the existing 1:1 `call.leave()`). `room.disconnect()` + stop local tracks + release `getUserMedia`. The Tauri close teardown gains a group-voice disconnect step (with the same 3s race-timeout backstop the 1:1 call uses). Critical ordering: voice teardown BEFORE Convex client teardown so `room.disconnect()`'s final signaling can land. The existing `onCloseRequested` + `handleLogout` structure is preserved ‚Äî Phase 6 adds the group-voice step, it doesn't restructure the teardown.
- **D13 ‚Äî No Convex auth; public token-mint action. v1 spoofing limitation accepted.** Inherits Phase 2 D7 / Phase 3 D4 / Phase 4 D7 / Phase 5 D7. The `mintToken(userId)` action is public (no Convex auth middleware); the client passes its own `userId`. **Known v1 limitation:** a misbehaving client could mint a token with a spoofed identity (pass someone else's `userId`) and join the voice room as that user. Acceptable for v1 (‚â§10 trusted friends, Convex dev backend, self-hosted LiveKit on a private VM); hardening (Convex auth gating the action, or signed identity) is deferred. Flagged in `validation.md`. The `mintToken` action loads the `users` doc for the passed `userId` to build name/metadata ‚Äî it does not verify the caller IS that user (that's the D13 gap).
- **D14 ‚Äî Platform scope: Windows-only for Phase 6 (inherits Phase 4 D9).** WebView2 mic access was validated in Phase 4. No new platform work; no macOS `NSMicrophoneUsageDescription` or Linux WebKitGTK handling. Cross-platform mic hardening defers.
- **D15 ‚Äî Validation: automated gates green + manual smokes.** Lint, typecheck, `bun tauri build`, Convex functions register (new `livekit.ts` action) must pass (inherited). Plus manual smokes for (i) 3+ join/leave freely, audio stable (the DoD); (ii) mute/deafen round-trip in group; (iii) leave + rejoin one-click, no teardown dance; (iv) side-by-side layout + roster + speaking indicators; (v) mutual exclusivity with 1:1 call (D9); (vi) coturn TURN fallback across NAT (conditional ‚Äî requires a strict-NAT test client); (vii) teardown on logout/window-close; (viii) no regression in Phase 2/3/4/5. See `validation.md`.
- **D16 ‚Äî No schema change.** Voice room state lives in LiveKit (D4), not Convex. No new tables, no new indexes, no migrations. The `conversations`/`messages`/`typing` tables are untouched (the lobby text half is complete from Phase 5). The `calls` table (Phase 4) is untouched (1:1 voice is complete). Cost: voice room state is not persisted/queryable from Convex (D4); acceptable for v1. Validated by `git diff convex/schema.ts` empty (the Phase-5 D1 gate, applied to Phase 6).
