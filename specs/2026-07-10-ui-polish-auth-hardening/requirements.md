# Phase 9 — UI Polish and Authentication Hardening: Requirements

Feature dir: `specs/2026-07-10-ui-polish-auth-hardening/`
Roadmap phase: **Phase 9 — UI Polish and Authentication Hardening** (`specs/roadmap.md`)
Mission ref: `specs/mission.md` · Stack ref: `specs/tech-stack.md`
Builds on: `specs/2026-07-10-post-production-0-1-0-fixes/` (Phase 8 — persistent session restoration, roomier window geometry, global voice controls, message actions, and non-blocking 1:1 call presentation that Phase 9 must preserve and finish hardening rather than reimplement independently).

## Goal
Phase 9 turns the feature-complete desktop shell into a cohesive, responsive, production-grade application. It uses the supplied Discord desktop screenshots as visual references while mapping only to BaatCheet's real loops: DMs, one shared lobby, one shared voice room, presence, settings, and existing calls/messaging. In parallel, it closes the current authentication and authorization gaps so sessions survive normal restarts without plaintext secrets and remote operations cannot impersonate another user.

## In scope
- Preserve the supplied screenshots under `references/discord-dm-reference.png` and `references/discord-lobby-reference.png`; maintain a UI inventory that maps each visible region to an existing BaatCheet feature or an explicit exclusion (Decisions D4, D5).
- Establish shared design tokens and primitives for panel colors, elevations, typography, spacing, radii, dividers, icon sizes, status colors, motion, focus rings, tooltips, menus, buttons, inputs, skeletons, notices, and destructive states.
- Recompose the shell into stable desktop regions: icon rail; context-sensitive left navigation; central DM/lobby content; and, in the lobby, an always-visible far-right member panel at every supported window size. Preserve the Phase 8 `960x640` minimum and keep the central conversation usable without horizontal page overflow (Decision D4).
- Redesign the DM surface from the first reference: compact DM navigation, clear active row, conversation header, Discord-style message rows/grouping, rich-message actions, composer, self controls, call controls, and polished empty/loading/error states.
- Redesign the lobby primarily from the second reference: a left sidebar with exactly one shared voice-channel item, central shared text thread, and far-right member panel grouped into Online and Offline using the existing reactive presence feed.
- Remove the lobby's top Join Voice button. Clicking the single sidebar voice item joins immediately, handles microphone permission/connection progress inline, expands participants beneath it, and exposes connection quality, mute, deafen, settings, and disconnect controls in the lower-left voice/user area (Decision D3).
- Preserve group/1:1 mutual exclusivity, background mute/deafen shortcuts, join/leave sounds, speaking indicators, teardown behavior, and one-click rejoin while changing presentation.
- Add responsive/accessibility behavior across the supported desktop range: predictable sizing, panel-owned scrolling, high-DPI and text resilience, visible keyboard focus, logical tab order, semantic labels, reduced-motion support, WCAG AA contrast, tooltips for icon-only controls, and no color-only state.
- Keep Windows Credential Manager as the primary credential authority. Replace `discord_tokens.json` plaintext storage with a versioned, integrity-protected Windows DPAPI CurrentUser fallback; migrate a valid legacy record once, verify the protected copy, and remove the plaintext source without logging it (Decision D1).
- Restore the last verified cached identity immediately from protected storage, render a clearly marked reconnecting/synchronizing state, and refresh Discord plus backend session state in the background. Gate remote privileged operations until server authentication is valid; transient network, Discord, Convex, or storage failures retain recoverable credentials and expose retry (Decision D2).
- Model auth states explicitly: checking local session, restored-local/synchronizing, authenticated-online, starting OAuth, waiting for callback, callback rejected/expired, retryable restore failure, revoked/needs login, and logging out. Make startup/listener behavior idempotent under React Strict Mode and duplicate deep-link delivery.
- Harden OAuth: external system browser only; Authorization Code + S256 PKCE; `identify` only; CSPRNG verifier/state; one pending attempt with creation time/expiry; single-use callback consumption; exact scheme/host/path; bounded unique query parameters; strict client/scope/token validation; refresh-token rotation; one refresh owner; finite timeouts; bounded response parsing; and safe retry/backoff.
- Pin the Discord application ID at the native boundary instead of accepting arbitrary renderer-supplied IDs. Validate all Tauri command and deep-link inputs by type, length, character set, multiplicity, and lifecycle before network or storage use.
- Remove token/profile response bodies, raw credential JSON, authorization codes, refresh tokens, access tokens, JWTs, and sensitive query strings from logs, panic files, errors, analytics, and frontend state. Use stable redacted error codes plus user-safe messages.
- Add a server-side Discord-to-BaatCheet auth bridge that verifies current Discord identity and issues short-lived, audience/issuer-bound JWTs accepted by Convex's custom JWT/OIDC configuration. Keep Discord refresh tokens native-only and BaatCheet access JWTs memory-only with proactive renewal.
- Replace unauthenticated `ConvexProvider` with an auth-aware provider. In every user-scoped Convex query/mutation/action, resolve the caller with `ctx.auth.getUserIdentity()`, map the immutable Discord subject to one `users` document, and authorize conversation membership, message ownership, presence ownership, call participation, lobby access, upload ownership, and LiveKit identity server-side (Decision D6).
- Remove caller authority from `userId`/`discordId` arguments where identity can be derived. Treat remaining resource IDs as untrusted input; validate membership, ownership, state transitions, size/length/count limits, URL/media constraints, and rate/abuse bounds before reads, writes, fetches, uploads, or token minting.
- Add focused Rust, React, and Convex tests plus hostile-client contract tests for OAuth callback replay/expiry/tampering, storage migration/corruption, restore/refresh races, log redaction, JWT expiry/audience/issuer/signature failures, cross-user reads/writes, upload and LiveKit mint authorization, and explicit logout.
- Conduct screenshot-based visual QA at the supported minimum, default, maximized, and common high-DPI sizes; record intentional differences caused by BaatCheet's smaller feature set rather than adding irrelevant Discord UI.

## Out of scope (deferred — explicitly NOT Phase 9)
- Servers, guild switching, text-channel hierarchies, additional voice channels, categories, roles, moderation, bots, integrations, threads, forums, events, quests, activities, Nitro, boosting, store/shop, gifting, or Discord account-management surfaces — not in v1.
- Video, screen sharing, streaming, recording, stage channels, push-to-talk, or voice-channel capacity tiers — not in this phase.
- Passwords, email login, passkeys, a second identity provider, or replacing Discord as the sole user-facing identity source — not in v1.
- Offline message sending, offline voice, or treating cached local identity as authorization for remote operations. Cached restore is presentation/continuity only until a valid backend token is available.
- Cross-platform encrypted fallback implementations beyond Windows DPAPI CurrentUser. Credential Manager remains primary; other desktop targets require a later platform-specific review.
- Claiming an external penetration test, third-party security certification, or production load sign-off from local automated checks. Those require independent evidence before public release.
- Phase 7's formal game-load performance measurements and production updater signing/hosting evidence; this phase must avoid regressions but does not mark those gates complete.

## Decisions (locked for this phase)
- **D1 — Credential Manager plus DPAPI fallback.** Windows Credential Manager remains primary. If it is unavailable, the fallback is a versioned blob encrypted and integrity-protected for the current Windows user with DPAPI; the existing plaintext JSON fallback is migrated and securely removed, never retained as a second copy.
- **D2 — Restore locally, then synchronize.** A protected cached profile may render the authenticated shell immediately with an explicit reconnecting/synchronizing state. It does not authorize remote work: authenticated Convex operations remain gated until Discord refresh/verification and a valid short-lived BaatCheet JWT are available; retryable failure preserves the protected session.
- **D3 — One-click shared voice join.** The lobby sidebar contains one voice-channel item and clicking it begins joining immediately. Connection progress, failure/retry, participants, connection status, mute, deafen, settings, and disconnect live inline in the left sidebar/lower-left controls; the obsolete top Join Voice button is removed.
- **D4 — Screenshot-led four-region lobby shell.** The second screenshot is the primary lobby reference: icon rail, left context sidebar, central text thread, and always-visible far-right members. The first screenshot guides DMs. Panel proportions may adapt to BaatCheet and the `960x640` minimum, but hierarchy, density, state language, and visual rhythm should remain recognizably Discord-derived.
- **D5 — Feature mapping, not Discord cloning.** Every visual element must map to a shipped BaatCheet capability. Blacked-out areas and unsupported Discord products are explicit exclusions; empty decorative replicas, dead buttons, fake channels, and placeholder commerce/activity UI are defects.
- **D6 — Server-verified identity and authorization.** Discord remains the sole identity provider, but a freshly verified Discord identity is exchanged for a short-lived BaatCheet JWT that authenticates Convex connections/RPCs. Convex derives the caller from `ctx.auth`; renderer-provided `userId`/`discordId` values never establish identity or permission.
- **D7 — Auth secrets stay native or server-side.** Discord refresh tokens remain in protected native storage, BaatCheet JWTs remain in memory, and signing keys remain server-side. localStorage, unencrypted files, frontend environment variables, logs, error strings, and Convex documents must never contain credential material.
- **D8 — Production readiness requires real evidence.** Automated tests are necessary but insufficient. Merge requires installed Windows restart/offline recovery tests, two-client authorization abuse tests, screenshot comparisons across supported sizes, keyboard/accessibility checks, and an explicit record of remaining external security/load evidence.

## Context
- `specs/mission.md` — the app serves one small friend group through three loops only; the redesign should make those loops familiar without importing Discord's server/channel/product overhead.
- `specs/tech-stack.md` — Tauri owns native OAuth/protected storage, React/Tailwind own the shell, Convex owns realtime data, LiveKit owns group voice, and Discord remains the sole identity provider.
- `specs/roadmap.md` Phase 8 — established persistent restore intent, roomier geometry, global voice controls, and non-blocking calls. Phase 9 audits the resulting implementation and closes security/UI gaps rather than assuming Phase 8 is sufficient.
- `specs/roadmap.md` Phase 9 DoD: *the installed Windows build matches the supplied references across supported lobby/DM/auth states without unsupported Discord features; the lobby retains its right member panel and one-click sidebar voice flow; repeated close/reopen and offline/online recovery preserve the session without plaintext credentials; OAuth replay/tampering and malformed inputs fail safely; and two-client hostile-call tests prove users cannot read, write, signal, upload, or mint tokens as another user.*
- Current UI state — `AuthenticatedLayout.tsx` composes an icon rail, collapsible left presence/DM sidebar, central content, optional left-side `VoiceStage`, and lobby header Join Voice action. Phase 9 moves group voice into the left sidebar and adds the right member panel without duplicating subscriptions or voice state.
- Current presence state — `presence.listPresence` already returns all known presence records sorted online-first with profile/status data, so the right panel can group Online/Offline without inventing a guild/member system.
- Current credential state — `store.rs` writes full Discord access/refresh tokens to both Credential Manager and plaintext `%APPDATA%/baatcheet/discord_tokens.json`, and logs raw stored JSON on deserialization failure. D1/D7 treat this as release-blocking secret exposure.
- Current OAuth state — S256 PKCE, random state, external browser, `identify` scope, and deep-link filtering exist. Missing hardening includes pending-flow expiry, bounded/unique callback parameters, client-ID pinning, response/time limits, strict token validation, replay/duplicate tests, and comprehensive redaction.
- Current API state — `src/main.tsx` uses unauthenticated `ConvexProvider`; public Convex functions accept `userId`/`discordId` as identity, and `presence.ts` explicitly documents spoofing as a trusted-client limitation. Phase 9 replaces that trust model across messages, calls, presence, typing, uploads, conversations, lobby access, and LiveKit minting.
- Standards references — Discord recommends validating unique OAuth `state`; RFC 8252 requires PKCE and an external browser for public native clients; Convex documents authenticated JWT connections plus `ctx.auth.getUserIdentity()`; Microsoft documents DPAPI CurrentUser protection and integrity checking. See [Discord OAuth2](https://discord.com/developers/docs/topics/oauth2), [RFC 8252](https://datatracker.ietf.org/doc/html/rfc8252), [Convex authentication](https://docs.convex.dev/auth/overview), [Convex custom JWT](https://docs.convex.dev/auth/advanced/custom-jwt), and [Windows DPAPI](https://learn.microsoft.com/en-us/windows/win32/api/dpapi/nf-dpapi-cryptprotectdata).

## User-performed prerequisites (not agent-executable)
- Confirm the production Discord application ID and exact `baatcheet://callback` registration in the Discord Developer Portal; no client secret is created or embedded.
- Provision production-only JWT signing material and issuer/audience/JWKS configuration for the self-hosted auth bridge/Convex deployment; secrets must be supplied through server environment variables, not committed.
- Provide two Discord test accounts plus an installed Windows release build for restart/offline/authorization/manual visual smokes.
- Make a Windows environment available where Credential Manager failure can be simulated safely so the DPAPI fallback and migration path can be proven.
- Arrange an independent security review/penetration test before describing the public release as externally security-validated.
