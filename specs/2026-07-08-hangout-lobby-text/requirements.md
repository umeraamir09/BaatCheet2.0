# Phase 5 — Hangout lobby (text half): Requirements

Feature dir: `specs/2026-07-08-hangout-lobby-text/`
Roadmap phase: **Phase 5 — Hangout lobby (text half)** (`specs/roadmap.md`)
Mission ref: `specs/mission.md` · Stack ref: `specs/tech-stack.md`
Builds on: `specs/2026-07-06-dm-text/` (Phase 3 — the generic `conversations`/`messages`/`typing` tables, the reactive subscription + public-mutation patterns, the 2-pane shell, and the DM-selectable friends sidebar), `specs/2026-07-06-presence/` (Phase 2 — presence + friends sidebar), and `specs/2026-07-07-voice-1-1/` (Phase 4 — the floating call overlay + DM-thread call button that must keep working alongside the lobby).

## Goal
The group text half of the centerpiece: one shared, always-on "lobby" room for the whole friend group. Any member posts to the lobby → all members see it live; history persists across restarts. This is the roadmap's Phase 5 and the text half of the "Talk to everyone" loop (`specs/mission.md`). The roadmap calls this "cheaply — reuses DM components," and Phase 3's forward-looking generic data model (Decision D1) was designed exactly for this: the lobby is a `conversations` doc with `type: "group"` reusing the same `messages` + `typing` tables — a UI retarget, not a schema migration. Phase 5 also lands the narrow icon rail that Phase 3 D2 deferred ("Phase 5 adds the icon rail when the group lobby becomes the second surface").

## In scope
- **Single shared lobby conversation (Decision D2):** exactly one `conversations` doc with `type: "group"`, `key: "group:lobby"`, looked up via the existing `byKey` index. The lobby is the whole group's shared space — no per-channel fragmentation (mission: "one single shared space for the whole group — no channel switching, no server list"). `participantIds` is maintained opportunistically (add self on first open) but is NOT an access gate — membership is implicit (everyone in `users` is a member). Auto-created on login (Decision D6) so the lobby is always-on.
- **No schema change (Decision D1):** the `conversations`/`messages`/`typing` tables from Phase 3 are already generic. `conversations.type` was `"dm"`; Phase 5 adds `"group"` values. `messages.conversationId` and `typing.conversationId` are already FK-to-`conversations` with no DM-specific coupling. This is the payoff of Phase 3 D1's forward-looking design — zero new tables, zero new indexes, zero migrations.
- **Lobby Convex functions (`convex/lobby.ts`, Decision D2):**
  - `getOrCreateLobby(userId)` mutation: look up by `key === "group:lobby"` (via `byKey`); if missing, insert `{ type: "group", participantIds: [userId], key: "group:lobby", createdAt: now, lastMessageAt: now }`; if found and `userId` not in `participantIds`, patch to add it; return the `conversationId`. Public (Decision D7).
  - `getLobby` query (reactive): return the single `conversations` doc where `key === "group:lobby"` (or `null` if it doesn't exist yet). Powers the reactive lobby-id subscription so the lobby appears as soon as it's created, with no polling.
- **Icon rail (Decision D3 — the deferred Phase-3 D2 surface):** a narrow (~56px) leftmost vertical rail with two icons: "Lobby" (group/hash icon) and "Direct Messages" (person icon). Selecting "Lobby" shows the group text thread in the main pane; selecting "DMs" shows the selected DM (existing Phase-3/4 behavior). The icon rail is always visible (not collapsible — it's the persistent navigation spine). The existing sidebar (friends/DMs/presence) persists in both modes — it is not hidden when the lobby is active. Clicking a friend/DM row in the sidebar while in "Lobby" mode switches to "DMs" mode and opens that DM (natural cross-navigation).
- **Chat hook generalization (Decision D4):** the existing `useDMThread` hook (`src/hooks/useDMThread.ts`) is already generic — it takes `conversationId` + `myUserId`, nothing DM-specific. Phase 5 renames it to `useChatThread` (`src/hooks/useChatThread.ts`) so both `DMThread` and `LobbyThread` use an honestly-named hook. This is a mechanical rename (file + export + interface name + imports), not new logic. The hook's reactive `listMessages` + `listTyping` + `sendMessage` + `setTyping` calls are unchanged — they're keyed on `conversationId`, which works for both DM and group.
- **LobbyThread component (`src/components/LobbyThread.tsx`, Decision D5):** a separate component (not a generalized DMThread) that reuses the message-list + composer + typing patterns but with a group-appropriate header:
  - Header: "Lobby" title + a group/hash icon (no peer avatar, no call button — group voice is Phase 6, Decision D8).
  - Message list: same bubble rendering as `DMThread` — own messages right-aligned, others left-aligned with sender avatar + name. The `MessageBubble` rendering is identical (every non-own message already shows sender avatar + name, which is essential in a multi-sender group context).
  - Typing indicator: multi-person format — 1 typer: "X is typing…"; 2 typers: "X and Y are typing…"; 3+: "X, Y and N others are typing…". The `listTyping` query already returns all typing peers (excluding self); the LobbyThread formats the text for N typers (the DMThread only shows `typingPeers[0]`).
  - Composer: identical to `DMThread`'s (textarea + Send; Enter sends, Shift+Enter newline; `notifyTyping` debounced ~300ms via the hook).
  - Empty state: "No messages yet. This is the group lobby — say hi to everyone!" (group-flavored vs. the DM's "Say hi!").
- **View mode state + persistence (Decision D10):** `AuthenticatedLayout` owns a `viewMode` state (`"lobby" | "dms"`). The last-selected mode persists across relaunch via localStorage (`baatcheet.viewmode`, a UI pref — same pattern as sidebar collapse + active DM). Cleared on logout (no cross-user leakage). Default on fresh login: `"lobby"` (the lobby is the centerpiece — land users there first).
- **Lobby auto-creation on login (Decision D6):** `AuthenticatedLayout` calls `getOrCreateLobby(userId)` once per session when `presence.userId` becomes available (ref-guarded to avoid re-calling on re-renders). This ensures the lobby doc exists for every user on every session. A reactive `getLobby` query provides the `conversationId` for the `LobbyThread` subscription. The lobby is always-on (roadmap: "Always-on, persisted history").

## Out of scope (deferred — explicitly NOT Phase 5)
- **Group voice / LiveKit SFU** → Phase 6 (the roadmap's "infra-heavy half"). Phase 5 is text-only; the `LobbyThread` has no call/join-voice button (Decision D8).
- **Side-by-side lobby layout (voice left, text right)** → Phase 6. Phase 5's lobby is a single text thread; the side-by-side layout that splits voice and text lands when group voice lands.
- **Multiple channels / rooms** → not in v1 (mission: "one single shared space — no channel switching, no server list"). The lobby is the one and only group room.
- **Message editing / deletion / reactions / attachments** → not in v1 (mission: pared-down; text only).
- **Read receipts / "seen" indicators** → not in v1.
- **Message search / full-text query** → not in v1.
- **Pagination / lazy-loading of lobby history** → not in v1 (mission YAGNI — same stance as Phase 3 D5; ≤10 group keeps the lobby small). The reactive subscription returns full history.
- **Lobby member roster panel** → the presence sidebar already shows who's online; a separate right-side member list is Discord chrome that the pared-down mission doesn't need. Deferred.
- **Offline push notifications for lobby messages** → out of scope for v1 (`specs/mission.md`).
- **Convex auth / per-user write authorization** → deferred (Decision D7 — public mutations are a known v1 limitation, inherited from Phase 3 D4).
- **Full Discord-derived theme polish** → Phase 7 (Phase 5 lobby UI is functional, lightly styled — reuses the Phase-3 dark theme tokens already in place).
- **Idle CPU/RAM profiling under gaming load** → Phase 7.

## Decisions (locked for this phase)
- **D1 — No schema change. The lobby is a UI retarget, not a migration.** Phase 3 D1 designed `conversations`/`messages`/`typing` to be generic (`conversations.type` = "dm" now, "group" later; `messages` + `typing` keyed on `conversationId` with no DM-specific fields). Phase 5 validates that design: the lobby is a `conversations` doc with `type: "group"`, `key: "group:lobby"`, reusing `messages` + `typing` verbatim. Zero new tables, zero new indexes, zero migrations. Cost: the `conversations` table now holds two `type` values (scanned by `listMyDMs` which already filters `type === "dm"` — the lobby is correctly excluded from the DM list). The alternative — a separate `groupMessages` table — would have been a migration and a second code path; Phase 3 D1 already rejected it.
- **D2 — Lobby identity: single doc, `key: "group:lobby"`, looked up via `byKey`.** The lobby is the one shared room for the whole group (mission: "one single shared space"). A fixed `key` (`"group:lobby"`) gives a single lookup path via the existing `byKey` index — no scan, no ambiguity. `participantIds` is maintained (add self on first open) but NOT an access gate — membership is implicit (the whole group). The alternative — a fixed `_id` — would bypass Convex's id generation and couple the lobby to a specific id string; the `key` approach is consistent with how DMs are looked up and is index-backed. The `key` `"group:lobby"` cannot collide with DM keys (`"userIdA__userIdB"` — two Convex ids joined by `__`).
- **D3 — Icon rail: the deferred Phase-3 D2 surface lands now.** Phase 3 D2 explicitly said "Phase 5 adds the icon rail when the group lobby becomes the second surface." The icon rail is a narrow (~56px) leftmost vertical bar with "Lobby" and "DMs" icons — the Discord-style DM-vs-server switcher, pared down to two destinations. The sidebar (friends/DMs/presence) persists in both modes (not hidden) — it's useful context in both views, and hiding it would lose the presence list while in the lobby. Clicking a friend/DM in the sidebar while in "Lobby" mode switches to "DMs" mode + opens that DM (natural cross-navigation — the sidebar rows are always interactive). The icon rail itself is NOT collapsible (it's the persistent navigation spine; the existing sidebar collapse toggle stays on the sidebar).
- **D4 — Reuse `useDMThread` (renamed `useChatThread`) for the lobby.** The hook is already generic — it takes `conversationId` + `myUserId` and calls `listMessages`/`listTyping`/`sendMessage`/`setTyping`, all keyed on `conversationId`. The rename (`useDMThread` → `useChatThread`, `UseDMThreadResult` → `UseChatThreadResult`, file `useDMThread.ts` → `useChatThread.ts`) is mechanical — no logic change. Both `DMThread` and `LobbyThread` import the renamed hook. The alternative — a separate `useLobbyThread` hook duplicating the same subscription logic — would be a copy-paste maintenance trap.
- **D5 — LobbyThread is a separate component, not a generalized DMThread.** The DM thread has peer-specific UI (peer avatar + name in the header, a call button gated on `peerOnline`, single-peer typing text). The lobby has group-specific UI (a "Lobby" title + group icon, no call button, multi-person typing text, a group-flavored empty state). Generalizing `DMThread` into a single component with conditional peer-vs-group branches would couple two distinct UIs and make both harder to read. A separate `LobbyThread` that reuses the same hook + the same bubble/composer rendering patterns is cleaner. Shared sub-components (`MessageBubble`, `Composer`) can be extracted if the duplication is bothersome, but the thread-level components stay separate.
- **D6 — Auto-create the lobby on login.** `getOrCreateLobby(userId)` is called once per session (in `AuthenticatedLayout`, ref-guarded) when `presence.userId` becomes available. This ensures the lobby doc always exists for every user — the lobby is always-on (roadmap: "Always-on, persisted history"). The alternative — lazy creation on first lobby click — adds a loading state (click → create → subscribe) and means the lobby might not exist for a user who never clicks it. Auto-creation is one extra mutation on login (alongside `upsertUser` + `setOnline`), negligible cost for ≤10 users. A reactive `getLobby` query provides the `conversationId` for the `LobbyThread` subscription — the lobby appears as soon as the doc lands.
- **D7 — No Convex auth in v1; public lobby mutations, keyed by `userId`.** Inherits Phase 3 D4 / Phase 2 D7 / Phase 1 D-impl-3. `getOrCreateLobby` is public (no Convex auth middleware); the client passes its own `userId`. **Known v1 limitation:** a misbehaving client could create a rogue lobby or add/remove participants. Acceptable for v1 (≤10 trusted friends, Convex dev backend); hardening deferred. Flagged in `validation.md`. `sendMessage` + `setTyping` are already public (Phase 3 D4) — the lobby reuses them unchanged.
- **D8 — No group voice in Phase 5. The LobbyThread has no call button.** Group voice via LiveKit SFU is Phase 6 (the roadmap's "infra-heavy half"). The `LobbyThread` header shows a "Lobby" title + group icon only — no call/join-voice button. 1:1 voice (Phase 4) continues to work from the DM thread; the floating call overlay persists across view-mode switches (a 1:1 call is independent of whether you're viewing the lobby or a DM).
- **D9 — Validation: automated gates green + manual smokes.** Lint, typecheck, `bun tauri build`, and Convex schema (no change expected — D1) must pass (inherited). Plus manual smokes for (i) post to lobby → all members see it live (the Phase-5 DoD, first half); (ii) history persists across restarts (the DoD, second half); (iii) typing indicator in group context (multi-person); (iv) navigation round-trip (lobby ↔ DMs, view-mode persistence); (v) no regression in Phase 2/3/4. See `validation.md`.
- **D10 — View mode persistence: last-selected mode (Lobby vs DMs) persists across relaunch.** Same localStorage pattern as sidebar collapse (`SIDEBAR_COLLAPSED_KEY`) and active DM (`ACTIVE_DM_KEY`) — a UI pref, not a credential. Cleared on logout (no cross-user leakage). Default on fresh login: `"lobby"` (the lobby is the centerpiece — land users there first, per the mission's "the centerpiece" framing).
